## Context

这次变更横跨认证、任务通知与项目目录选择三条链路：
- 认证侧已有多用户设计，但仍保留 `default` 空密码兼容逻辑。
- 通知侧目前完全内嵌在任务记录中，所有 webhook 与 secret 都按任务冗余存储。
- 目录选择器已有搜索和浏览能力，但缺少“就地创建目录”。

其中通知模板抽离会引入新的持久化对象和任务引用关系，因此需要明确设计。

## Goals

- 登录必须依赖显式账号密码，避免匿名或空密码进入系统
- 通知配置可复用、集中维护，任务层只做引用
- 目录选择流程支持新建目录，减少跳出 PromptX 到系统文件管理器的操作

## Non-Goals

- 不引入复杂角色权限系统
- 不实现目录重命名、批量管理或删除
- 不改动现有 Relay 登录体系

## Design Notes

### 认证

- 首次启动不再自动创建 `default` 空密码用户
- 如果用户配置为空，系统应提示管理员先通过 CLI 添加账号后再登录
- 登录校验必须拒绝空密码的默认兼容路径

### 通知模板

- 新增独立的通知模板存储，字段覆盖当前任务通知里的通道类型、webhook、secret、触发条件、语言与消息模式
- 任务侧改为保存模板引用与少量运行态结果，而不再持有完整 webhook 配置
- 需要提供模板列表/创建/更新/删除接口与前端选择器
- 迁移策略可采用：
  - 保留老任务里的旧通知配置，作为兼容读取来源
  - 新编辑后的任务逐步切到模板引用

### 目录选择器

- 在目录选择弹窗中增加“新建目录”操作
- 创建目录时基于当前浏览中的父目录或当前输入路径创建
- 服务端需要负责目录名校验与安全边界控制，避免越界创建
